题目	基于Django的宿舍报修工单系统的设计与实现


第一部分：系统详细设计 (System Design)
为了支撑你的毕业论文，我们需要把系统拆解得非常细致。

1. 总体架构 (Architecture)
设计模式： 采用 Django 标准的 MTV (Model-Template-View) 模式。

前后端交互：

前端： HTML5 + CSS3 (Bootstrap 5) + JavaScript (jQuery/Ajax)。

亮点建议： 利用你的UI优势，引入 LayUI 或 SimpleUI 来美化后台，拒绝原生Django Admin的“土气”。

后端： Python 3.x + Django 4.x。

数据库： MySQL 8.0。

2. 数据库设计 (Database Modeling)
这是论文的核心，直接关系到你后期的代码好不好写。建议设计以下 5 张核心表：

A. 用户信息表 (Users)

继承 Django 的 AbstractUser。

role: 身份标识（1:学生, 2:维修人员, 3:宿管/管理员）。

phone: 手机号。

dorm_code: 宿舍号（如 "北一-305"）。

avatar: 头像（提升UI体验）。

B. 报修工单表 (WorkOrder) —— 最核心的表

id: 自增主键。

order_sn: 工单编号（唯一，生成规则：YYYYMMDD+随机码，体现专业性）。

user_id: 外键，关联学生。

type_id: 外键，关联故障类型。

status: 状态机（0:待审核, 1:已派单, 2:维修中, 3:已完成, 4:已取消）。

content: 故障描述。

img_proof: 现场照片（支持上传路径）。

repairman_id: 外键，关联维修人员（可为空，派单后填充）。

create_time: 提交时间。

finish_time: 完工时间。

C. 故障类型表 (RepairType)

name: 类型名（如：水电类、家具类、门窗类、网络类）。

priority: 优先级（高/中/低，用于后台排序显示紧急程度）。

D. 工单日志/流转记录表 (OrderLog)

设计目的： 实现“可追溯”。记录每一个状态变化的瞬间。

order_id: 关联工单。

operator: 操作人（是谁派的单？是谁点的完工？）。

action: 动作（提交、审核、派单、完工、评价）。

remark: 备注（如：“缺少零件，需采购”）。

create_time: 操作时间。

E. 评价表 (Comment)

order_id: 关联工单。

score: 评分（1-5星）。

feedback: 文字反馈。

3. 核心业务逻辑 (Business Logic)
在开题报告或论文中，你需要描述以下三个核心流程：

流程一：学生报修（前端逻辑）

登录 -> 进入“我的报修” -> 点击“新增”。

填写表单（自动回填宿舍号，上传图片时进行JS压缩）。

提交 -> 后端生成 order_sn -> 写入数据库 -> 状态置为 0 (待审核)。

流程二：派单逻辑（管理员/自动）

管理员进入“工单池” -> 筛选 status=0 的工单。

点击“派单” -> 弹出模态框选择空闲的维修员。

系统动作： 更新工单 status=1，写入 repairman_id，同时在 OrderLog 表插入一条派单记录。

流程三：维修闭环（维修员）

维修员登录 -> 看到“我的待办”。

点击“开始维修” -> 状态变 2 (维修中)。

点击“完工” -> 填写维修耗材/备注 -> 状态变 3 (已完成)。

系统自动触发通知给学生（模拟）。

一、选题依据及意义 (Refined)
(替换原稿，强调“数字化”和“数据价值”) 随着高校后勤管理数字化转型的推进，传统的人工报修模式（纸质登记、电话报修、微信群喊话）已暴露出**“信息孤岛严重、响应滞后、过程不可追溯、服务质量难以量化”**等弊端。 本研究的意义在于：

工程应用价值： 基于Django框架构建一套标准化的工单流转系统，实现从“报修-派单-维修-评价”的全生命周期闭环管理，解决报修信息不对称的问题。

数据价值： 通过系统积累的维修数据，可以形成“后勤大数据”，为学校统计设施故障率、优化耗材采购、考核维修人员绩效提供数据支撑，实现由“被动维修”向“主动维护”的转变。

技术实践意义： 本项目将软件工程的规范化流程（需求分析、系统设计、编码实现、软件测试）应用于实际场景，探索Web开发中高并发处理、权限控制（RBAC）及工作流设计的最佳实践。

二、研究目标与主要任务 (Refined)
(增加技术深度词汇) 研究目标： 设计并实现一个高可用、易扩展的B/S架构宿舍报修系统。系统需具备RBAC权限控制、可视化数据看板及响应式交互体验。

主要任务：

需求分析与建模： 使用UML（统一建模语言）绘制用例图、时序图和E-R图，明确系统的功能边界与数据结构。

系统架构搭建： 搭建Django MTV框架，集成 SimpleUI 优化后台管理界面，设计RESTful风格的内部接口。

核心模块开发：

工单流转引擎： 实现基于状态机的工单处理逻辑（提交→审核→派单→维修→验收→评价）。

权限管理模块： 基于Django Auth模块实现学生、维修员、管理员的三级权限隔离。

数据可视化模块： 引入 ECharts 图表库，动态展示“本月维修完成率”、“故障类型分布饼状图”等关键指标。

系统测试与部署： 编写单元测试用例，进行功能测试与压力测试；使用Nginx+uWSGI进行模拟部署环境搭建。

三、研究方法和手段 (Refined)
(让方法论看起来更科学)

文献研究法： 查阅国内外关于高校后勤信息化、Django框架应用、工作流引擎设计的相关文献，为系统架构设计提供理论依据。

软件工程方法： 采用 敏捷开发（Agile） 模式进行迭代。

设计阶段： 使用Axure RP制作高保真原型图，确认交互逻辑。

开发阶段： 采用前后端不完全分离模式（Django Templates），利用Ajax技术实现页面无刷新操作，提升用户体验。

数据库设计： 遵循数据库第三范式（3NF），确保数据冗余最小化。

对比实验法： 在系统完成后，选取部分宿舍进行试点，对比使用系统前后的“平均报修响应时间”和“学生满意度”，用量化数据验证系统的有效性。
